# 프로젝트 유형 분기 가이드

> **목적**: Wave Orchestration 파이프라인 진입 시 프로젝트 유형에 따라 적절한 온보딩 경로로 안내합니다.
> **대상**: PM, 기획자, Orchestrator Agent
> **버전**: 1.0

## 🚀 빠른 시작

프로젝트를 시작하기 전, 아래 질문에 답해주세요:

```
┌─────────────────────────────────────────────────────────────┐
│  🎯 프로젝트 유형을 선택해주세요                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [A] 🌱 신규 프로젝트 (0부터 시작)                           │
│      → 아직 코드가 없음                                      │
│      → 디자인 시스템부터 결정 필요                           │
│      → 기술 스택 선택 필요                                   │
│                                                             │
│  [B] 🔧 기존 프로젝트 (유지보수/신규 기능)                    │
│      → 이미 운영 중인 코드베이스 존재                        │
│      → 기존 디자인 시스템/기술 스택 활용                     │
│      → 새로운 기능 추가 또는 버그 수정                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘

선택: [A] 또는 [B]
```

## 🌱 [A] 신규 프로젝트 경로

### Phase 구조

```
Phase 1: 디자인 시스템 결정 (30분) ⭐ 신규 프로젝트만
├─ Design System Agent 투입
├─ 비기술적 질문으로 니즈 파악
├─ 스크린샷 분석 (선택)
└─ 오픈소스 디자인 시스템 추천

Phase 2: 프로젝트 설계 (1.5시간)
├─ PRD, RFC, Orchestration 문서 작성
├─ 기술 스택 결정 (디자인 시스템 기반)
└─ 스키마 및 폴더 구조 설계

Phase 3+: 개발 (병렬화 시작)
└─ 인프라 → 핵심 기능 → UI 완성 → 테스트 → 배포
```

### 상세 가이드

**→ [AGENT-ONBOARDING-GREENFIELD.md](./AGENT-ONBOARDING-GREENFIELD.md) 참조**

### 핵심 포인트

| 항목 | 설명 |
|------|------|
| 첫 번째 Agent | Design System Agent |
| 디자인 시스템 | 비기술적 질문 + 스크린샷 분석으로 추천 |
| 기술 스택 | 디자인 시스템 기반 자동 결정 |
| 병렬화 시작 | Phase 4부터 |
| 예상 시간 절약 | 30-35% |

## 🔧 [B] 기존 프로젝트 경로

### Phase 구조

```
Phase 0: 설계 (45분-1.5시간)
├─ 기존 코드베이스 분석
├─ 변경 범위 파악
└─ 에이전트 할당 계획

Phase 1+: 개발 (즉시 병렬화 가능)
└─ 기능 구현 → 통합 → 테스트 → 배포
```

### 상세 가이드

**→ [AGENT-ONBOARDING-EXISTING.md](./AGENT-ONBOARDING-EXISTING.md) 참조**

### 핵심 포인트

| 항목 | 설명 |
|------|------|
| 첫 번째 Agent | Architect/Analyzer (기존 코드 분석) |
| 디자인 시스템 | 기존 시스템 그대로 활용 |
| 기술 스택 | 기존 스택 유지 |
| 병렬화 시작 | Phase 1부터 |
| 예상 시간 절약 | 40-50% |

## 📊 프로젝트 유형 비교표

| 구분 | 신규 프로젝트 (A) | 기존 프로젝트 (B) |
|------|-------------------|-------------------|
| **코드베이스** | 없음 | 있음 |
| **디자인 시스템** | 선택 필요 (Phase 1) | 기존 활용 |
| **기술 스택** | 결정 필요 (Phase 2) | 기존 유지 |
| **첫 Phase** | 디자인 시스템 결정 | 코드베이스 분석 |
| **병렬화 시점** | Phase 4+ | Phase 1+ |
| **총 Phase 수** | 7개 (표준) | 4-5개 (표준) |
| **시간 절약** | 30-35% | 40-50% |
| **주요 리스크** | 스키마 불일치 | 기존 코드 호환성 |

## 🔄 하이브리드 경로 (특수 케이스)

기존 프로젝트에서 디자인 시스템만 교체하는 경우:

1. **Phase 1**: GREENFIELD의 디자인 시스템 결정 프로세스 수행
   - Design System Agent가 새 시스템 추천
   - 기존 코드와의 호환성 검토 추가

2. **Phase 2+**: EXISTING 프로세스로 진행
   - 기존 코드베이스 분석
   - 디자인 시스템 마이그레이션 계획
   - 점진적 교체 (컴포넌트 단위)

## 🤖 Orchestrator를 위한 자동 분기 로직

```javascript
// 자동 분기 의사결정 로직
function routeProject(projectInfo) {
  // 1. 코드베이스 존재 여부 확인
  if (!projectInfo.hasExistingCodebase) {
    return {
      route: 'GREENFIELD',
      startPhase: 1,  // 디자인 시스템 결정
      document: 'AGENT-ONBOARDING-GREENFIELD.md'
    };
  }

  // 2. 디자인 시스템 교체 여부 확인
  if (projectInfo.replaceDesignSystem) {
    return {
      route: 'HYBRID',
      startPhase: 1,  // 디자인 시스템 결정
      document: 'AGENT-ONBOARDING-GREENFIELD.md',
      switchToExisting: 'Phase 2'
    };
  }

  // 3. 기존 프로젝트 경로
  return {
    route: 'EXISTING',
    startPhase: 0,  // 기존 코드 분석
    document: 'AGENT-ONBOARDING-EXISTING.md'
  };
}
```

## 📋 체크리스트

### 신규 프로젝트 시작 전 확인사항

- [ ] 프로젝트 목표가 명확한가?
- [ ] 타겟 사용자가 정의되었나?
- [ ] 참고할 시각적 레퍼런스가 있나? (선택)
- [ ] MVP 범위가 정해졌나?
- [ ] 예상 일정이 있나?

### 기존 프로젝트 시작 전 확인사항

- [ ] 코드베이스 접근이 가능한가?
- [ ] 기존 아키텍처 문서가 있나?
- [ ] 변경 요구사항이 명확한가?
- [ ] 테스트 환경이 준비되었나?
- [ ] 배포 프로세스가 정립되었나?
