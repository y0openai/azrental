# 🎯 오케스트레이션 모드 선택 가이드

> **목적**: PM이 병렬 vs 순차 모드를 선택하도록 도움
> **대상**: 프로젝트 관리자 (PM)
> **읽는 시간**: 5분
> **언어**: 한글

---

## 🤔 어떤 모드를 선택할까?

이 간단한 결정 트리를 따라하세요:

---

## 📋 선택 기준

### 1️⃣ **개발 시간이 얼마나 필요한가?**

```
필요 시간 ┐
         │  ┌─ 3일 이상 → Mode 1 추천
         │  │
      2-3일 ┤  ← Mode 1 또는 2 모두 가능
         │  │
      1-2일 ┤  ├─ Mode 2 추천
         │  │
         └──┘
```

**기준**:
- **1-2일**: Mode 2 (순차) 추천 → 간단한 기능
- **2-3일**: 둘 다 가능 → PM이 선택
- **3-5일 이상**: Mode 1 (병렬) 추천 → 복잡한 기능

---

### 2️⃣ **에이전트 리소스가 충분한가?**

```
Mode 1 (병렬):
- Backend Agent ✅
- Frontend Core Agent ✅
- Frontend UI Agent ✅
- QA Agent ✅
→ 4명 모두 필요

Mode 2 (순차):
- 1명의 다재다능한 에이전트 ✅
→ 1명만 필요
```

**질문**:
- "4명의 에이전트를 동시에 실행할 수 있나?" → YES → Mode 1 가능
- "1명만 사용 가능한가?" → YES → Mode 2 필수

---

### 3️⃣ **기능의 복잡도는?**

#### **간단한 기능 (Mode 2 추천)**
- 예: "로그인 화면 개선"
- 예: "검색 기능 추가"
- 특징: 백엔드 OR 프론트엔드, 둘 다는 아님

#### **중간 복잡도 (Mode 1 또는 2)**
- 예: "새로운 경험 리스트 페이지 구축"
- 특징: 백엔드 + 프론트엔드 모두 필요
- 선택: PM이 결정

#### **복잡한 기능 (Mode 1 추천)**
- 예: "신규 신뢰 & 안전 시스템"
- 예: "크레딧 시스템 리뉴얼"
- 특징: 복잡한 API + 여러 UI 페이지 + 테스트
- 팀이 필요: 4명의 에이전트가 협력하면 빠름

---

## ✅ 의사결정 체크리스트

**다음을 확인하고 선택하세요**:

### Mode 1: 병렬 개발 체크리스트

- [ ] 개발 시간이 3일 이상이다
- [ ] Backend Agent를 실행할 수 있다
- [ ] Frontend Core Agent를 실행할 수 있다
- [ ] Frontend UI Agent를 실행할 수 있다
- [ ] QA Agent를 실행할 수 있다
- [ ] 복잡한 기능이다 (다중 도메인)

**모두 체크 되면**: ✅ **Mode 1 선택**

---

### Mode 2: 순차 개발 체크리스트

- [ ] 개발 시간이 1-2일이다
- [ ] 에이전트 리소스가 제한적이다
- [ ] 간단한 기능이다 (백엔드 또는 프론트엔드, 둘 다는 아님)
- [ ] PM의 조정이 최소 필요하다

**대부분 체크 되면**: ✅ **Mode 2 선택**

---

## 📊 실제 예시

### 예시 1: "Login 페이지 개선"

```
분석:
- 예상 시간: 1-2일
- 필요: Backend (로그인 API 개선) + Frontend (UI 개선)
- 복잡도: 낮음 (간단한 API)

선택: Mode 2 (순차 개발)
이유: 간단하고 빠르게 끝남

실행:
1. Backend Agent가 로그인 API 개선
2. Frontend Agent가 로그인 UI 개선
3. QA Agent가 통합 테스트
```

---

### 예시 2: "신규 크레딧 시스템"

```
분석:
- 예상 시간: 5-7일
- 필요: Backend (크레딧 로직) + Frontend (구매 UI, 잔액 표시) + QA (복잡한 테스트)
- 복잡도: 높음 (여러 도메인, 보안 이슈)

선택: Mode 1 (병렬 개발)
이유: 복잡하고 빠른 개발이 필요

실행:
Day 1: 문서 읽기, 역할 할당
Day 2-3:
  - Backend: 크레딧 로직 TDD
  - Frontend Core: 크레딧 Hook 개발
  - Frontend UI: 구매/잔액 UI 개발
  - QA: 테스트 시나리오 준비
Day 4-7: Mock 제거, 통합, 최종 테스트
```

---

### 예시 3: "관심사 매칭 개선"

```
분석:
- 예상 시간: 3-4일
- 필요: Backend (매칭 알고리즘) + Frontend (필터 UI, 결과 표시)
- 복잡도: 중간 (알고리즘은 복잡하지만 UI는 간단)

선택: Mode 1 (병렬 개발)
이유: 3일 이상이고 백엔드+프론트엔드 모두 필요

실행:
Day 1: 문서 읽기, 역할 할당
Day 2-3:
  - Backend: 매칭 알고리즘 TDD
  - Frontend: 필터 UI + 결과 표시
  - QA: 테스트 시나리오
Day 4-5: 통합, 최종 테스트
```

   - `03-ORCHESTRATION.md`의 "순차 개발 명령어" 섹션 수행
   - 1개 탭 준비

   - 리소스 절약 필요 → Mode 2

## 💡 자주 묻는 질문

### Q1: "Mode 1이 항상 더 빠른가?"

**A**: 네, 하지만:
- Mode 1: 3-5일 (병렬)
- Mode 2: 5-7일 (순차)
- 차이: 2-3일 정도

작은 기능이면 Mode 2도 충분합니다.

---

### Q2: "Mode를 중간에 바꿀 수 있나?"

**A**: 가능하지만 권장하지 않습니다.
- 문서, 역할 할당이 다르기 때문
- 처음부터 다시 시작해야 할 수 있음

**권장**: 시작 전에 신중히 선택하기

---

### Q3: "복잡도를 어떻게 판단하나?"

**A**: 다음을 확인하세요:
- 백엔드 변경 필요: ✅
- 프론트엔드 변경 필요: ✅
- 새로운 알고리즘: ✅
- 보안/권한 관련: ✅
- 테스트 복잡: ✅

**체크 3개 이상** → 복잡한 기능 → Mode 1 추천

---

### Q4: "프로토타입은 어떤 모드?"

**A**: Mode 2 (순차)
- 빠르게 구현하고 테스트
- 나중에 Mode 1로 전환 가능
